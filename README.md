Крок 1: Клієнт ➡️ API Gateway (HTTP-рівень)

    Мобільний додаток відправляє HTTP POST запит на адресу http://api.твоя-система.com/order.

    У заголовках (Headers) передається токен: Authorization: Bearer <JWT_TOKEN>.

    У тілі запиту (Body) йде JSON: {"product_id": 77, "quantity": 1}.

    API Gateway слухає цей порт. Його HTTP-сервер (наприклад, фреймворк Gin) приймає цей запит.

Крок 2: Усередині API Gateway (Перевірка токена)

Перш ніж створювати замовлення, гейтвей повинен переконатися, що користувач справжній. За це відповідає файл pkg/auth/middleware.go.

    Middleware перехоплює запит і витягує з заголовка рядок <JWT_TOKEN>.

    Гейтвей відкриває файл pkg/auth/client.go. Там ініціалізовано gRPC-клієнт, який тримає зв'язок із сервісом авторизації по внутрішньому порту (наприклад, localhost:50051).

    Гейтвей викликає gRPC-метод ValidateToken(), передаючи туди цей рядок.

    Запит трансформується з HTTP/JSON у бінарний протокол gRPC (HTTP/2) і летить у перший мікросервіс.

Крок 3: Робота go-grpc-auth-svc (Сервіс авторизації)

    Сервіс авторизації приймає gRPC-запит на порту 50051.

    Його логіка розшифровує JWT-токен за допомогою секретного ключа.

    Сервіс перевіряє: чи не закінчився термін дії токена? Якщо токен валідний, він дістає з нього інформацію про користувача (наприклад, User_ID: 99).

    Сервіс auth-svc відправляє gRPC-відповідь назад у Гейтвей: Status: OK, User_ID: 99.

Крок 4: Повернення в API Gateway та перенаправлення запиту

    Гейтвей отримує відповідь від auth-svc. Оскільки статус OK, middleware.go дозволяє запиту йти далі.

    Тепер керування переходить до файлу pkg/order/routes/create_order.go.

    Цей обробник бере User_ID: 99 (який ми щойно дізналися), бере product_id: 77 і quantity: 1 з початкового JSON.

    Гейтвей відкриває файл pkg/order/client.go (gRPC-клієнт до сервісу замовлень, наприклад, порт 50053).

    Гейтвей робить новий gRPC-виклик — метод CreateOrder(), передаючи туди ці три параметри. Запит знову кодується в gRPC і летить у сервіс замовлень.

Крок 5: Робота go-grpc-order-svc (Сервіс замовлень)

Тепер запит знаходиться всередині мікросервісу замовлень. Йому потрібно виконати бізнес-логіку.

    Сервіс замовлень приймає запит на створення замовлення для User_ID: 99 на товар Product_ID: 77.

    Але сервіс замовлень не знає, чи є такий товар і скільки він коштує! У нього немає доступу до бази даних товарів.

    Тому go-grpc-order-svc сам виступає як gRPC-клієнт. Він робить gRPC-виклики прямо в третій мікросервіс — go-grpc-product-svc (наприклад, порт 50052), викликаючи метод DecreaseStock() (зменшити кількість на складі).

Крок 6: Робота go-grpc-product-svc (Сервіс товарів)

    Сервіс товарів приймає gRPC-запит на порту 50052.

    Він робить SQL-запит до своєї власної бази даних (наприклад, PostgreSQL для товарів):
    SQL

    SELECT stock, price FROM products WHERE id = 77;

    База даних відповідає: на складі є 5 штук, ціна $500.

    Сервіс перевіряє: 5 > 1 (товар є). Він оновлює базу даних, зменшуючи кількість:
    SQL

    UPDATE products SET stock = stock - 1 WHERE id = 77;

    Сервіс товарів повертає gRPC-відповідь у сервіс замовлень: Status: Success, Price: 500.

Крок 7: Завершення замовлення в go-grpc-order-svc

    Сервіс замовлень отримав відповідь від сервісу товарів: товар успішно списано, ціна отримана.

    Тепер сервіс замовлень робить SQL-запит до своєї власної бази даних замовлень:
    SQL

    INSERT INTO orders (user_id, product_id, quantity, total_price, status) 
    VALUES (99, 77, 1, 500, 'CREATED');

    База даних зберігає замовлення і повертає його номер, наприклад, Order_ID: 4567.

    Сервіс замовлень віддає gRPC-відповідь назад в API Gateway: Order_ID: 4567, Status: Created.

Крок 8: API Gateway ➡️ Клієнт (HTTP-відповідь)

    Обробник у гейтвеї (pkg/order/routes/...) отримує фінальну gRPC-відповідь від сервісу замовлень.

    Він трансформує цей бінарний gRPC-пакет назад у звичайний текст HTTP JSON.

    Формується HTTP-відповідь зі статусом 201 Created і тілом:
    JSON

    {
      "order_id": 4567,
      "status": "Created",
      "message": "Замовлення успішно оформлено"
    }

    Мобільний додаток користувача отримує цей JSON і малює на екрані зелену галочку "Успішно!".
